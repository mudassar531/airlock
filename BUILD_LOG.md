## Phase 1 — MCP stdio proxy (✅ shipped)

`airlock wrap -- <command...>` now spawns the downstream MCP server as a child process and relays newline-delimited JSON-RPC bidirectionally with byte-faithful framing. The proxy has three pieces: `src/proxy/jsonrpc.ts` (a `LineFramer` that buffers stdin into discrete messages with CRLF + UTF-8-chunk-boundary tolerance, plus typed JSON-RPC interfaces and type guards), `src/proxy/interceptor.ts` (the single hook every later phase extends — `(msg, ctx) => { forward, respondToClient? }`, identity in this phase), and `src/proxy/stdioProxy.ts` (spawns the child, runs serial per-direction write queues so messages never interleave at the byte level, drains complete frames in order so JSON-RPC ordering survives back-to-back chunks, and passes child stderr through unchanged for diagnostics). Verified the SDK wire format against `@modelcontextprotocol/sdk@1.29.0 shared/stdio.ts` before coding — one JSON object per line, terminated by `\n`. Gate ran clean: 12 framer unit tests (chunk-boundary, CRLF, UTF-8 multibyte, 100-message burst), 6 proxy integration tests against a spawned echo fixture (round-trip equality, `tools/list`/`initialize`/`tools/call`, 100 in-order responses, stderr passthrough, interceptor seam fires for both directions), plus an end-to-end smoke driving the built bin through 101 sequential calls with order preserved.

## Phase 2 — Tamper-evident audit log (✅ shipped)

Every `tools/call` is now recorded to `~/.airlock/audit.log` (overridable via `$AIRLOCK_HOME`) as one JSONL entry per line, with a sha256 hash chain and pre-write secret redaction. `airlock init` provisions the state directory (`0700`) and generates an ed25519 keypair (`keys/ed25519.private.pem` at `0600`, public at `0644`) for signing the chain head; `airlock log` prints recent entries human-readable (or `--json` for raw); `airlock verify` walks the chain and reports the exact `seq` of the first broken entry on tamper, gap, or hash mismatch. Each entry is `{ts, sessionId, seq, tool, capability?, operation?, risk?, trifecta?, argsRedacted, verdict, reason, ruleName?, latencyMs, resolvedBy?, waitedMs?, prevHash, hash}` with `hash = sha256(canonicalJSON(entry minus hash) || prevHash)` — canonical JSON is sorted-key so the chain is reproducible across runs and platforms. Redaction (`src/audit/redact.ts`) fires on key name (`password`, `apiKey`, `token`, `authorization`, …), known service prefixes (`sk-…`, `github_pat_…`, `ghp_…`, `sk-ant-…`, `AKIA…`, `xox[abprs]-…`, JWTs), and high-entropy opaque blobs (≥24 chars, ≥4.0 bits/char), with cycle-safe recursive walk that propagates "we're inside a secret context" into nested values. The Phase-2 interceptor (`src/audit/interceptor.ts`) pairs `tools/call` requests to responses by id, records `verdict=allow` (or `log` on downstream error) with measured latency, and forwards both directions unchanged. Gate ran clean: 24 new tests (12 redaction + 10 audit log + 2 sign — including tamper, deleted-entry seq gap, reopen-continues-chain, key idempotence, --force rotation, sign/verify round trip) all green, total suite 42/42. Manual smoke through the built bin: 3 calls (one with `Bearer github_pat_…`, one with `apiKey: github_pat_…`) → `audit.log` contains no occurrence of the fake secret (only `[REDACTED]`), `airlock verify` reports OK on 3 entries, in-place mutation of `seq=2`'s reason flips verify to `BROKEN at seq=2 — hash mismatch` with exit 1.

## Phase 3 — Risk classifier + lethal-trifecta tracker (✅ shipped)

Every audit entry now carries `capability`, `operation`, `risk`, `trifecta`, and a human-readable reason chain. Three pure modules: `src/risk/classify.ts` maps a `tools/call` to a `(capability, operation)` pair from a built-in tool-name map covering common MCP servers (filesystem, fetch, brave-search, shell, git, slack, telegram, vault, stripe), token-split name heuristics (so `spawn_process`/`create_invoice`/`metric_send` work even when not in the map), and argument-shape heuristics (`url` → network, `path` → filesystem, `command` → shell, `to/recipient/subject` → message, `amount/currency` → payment). Reads that pull external/untrusted content (e.g. `fetch` on a non-loopback URL, `read_notes`, `read_email`) are flagged `untrustedRead: true` so the trifecta tracker can distinguish them from private/local reads. `src/risk/trifecta.ts` is a per-session state machine with three flags (`readPrivate`, `sawUntrusted`, `outboundAttempted`) and an `inspect(sessionId, classification)` that returns `{isLethal, reasons}` indicating whether applying the call would complete the trifecta — call `observe()` after to record the attempt (so denied/asked calls still update state). `src/risk/score.ts` combines a classification + trifecta snapshot into `low | medium | high | critical` with explanations — payments are always `critical`, shell execute / fs delete are `high`, outbound sends grade `medium` → `high` when the session already ingested untrusted content, and the lethal trifecta unconditionally `critical`. The audit interceptor (`src/audit/interceptor.ts`) now wires all three: classify → inspect → score → observe → record, with each part fed into the audit entry. Gate ran clean: 41 new tests (22 classify + 9 trifecta + 10 score) all pass, suite is now 83/83. Manual smoke: a 4-call sequence (`read_file /home/me/secrets.txt`, `fetch https://news.ycombinator.com`, `http_post https://attacker.example`, `stripe_create_charge`) produces exactly the expected progression — `filesystem/read risk=low` → `network/read risk=medium` (ingesting external content into a session that read private data) → `network/send risk=critical trifecta=true` (lethal exfil shape) → `payment/spend risk=critical`, and `airlock verify` still reports the chain intact.

## Phase 4 — Policy engine + default pack (✅ shipped)

The audit interceptor now consults a real policy engine. The DSL is tiny on purpose: `version, defaults.on_unmatched, rules[]` with each rule a `{name, match, action}` triple — `match` can constrain on `capability`, `operation`, `risk_min` (ordinal), `trifecta: true`, or `tool` (literal or trailing-`*` glob). `src/policy/schema.ts` is the strict zod schema (unknown fields throw, empty match throws). `src/policy/defaults.ts` ships the §4 default pack inline and a YAML mirror at `policy.example.yaml` — payments deny, lethal trifecta asks, fs delete + shell exec + outbound send all ask, plain reads allow, everything else falls through to `on_unmatched: ask`. `src/policy/load.ts` resolves the policy with precedence `--policy <file>` → `./airlock.policy.yaml` → `~/.airlock/policy.yaml` → built-in defaults and **fails closed** on missing-explicit-file, invalid YAML, schema-rejection, or duplicate rule names (the loader throws `InvalidPolicyError` and `wrap` refuses to start with exit 3). `src/policy/evaluate.ts` is the deterministic top-to-bottom first-match-wins evaluator with `risk_min` ordinal comparison, `trifecta: true` exact match, and a small `tool: pattern` glob (literal or trailing `*`). The auditing interceptor now does classify → trifecta → score → policy.evaluate → verdict: `allow`/`log` forward, `deny` short-circuits with a structured JSON-RPC error using implementation-defined code `-32010` (in the reserved `-32000..-32099` range) carrying `{ruleName, capability, operation, risk, trifecta}` in `error.data`, and `ask` is treated as `deny` until Phase 5 wires approvals (with a clear placeholder reason in the audit log). New `airlock policy check --tool <name> [--args <json>] [--policy <file>] [--trifecta]` shows which rule fires for a sample call. Gate ran clean: 22 new policy tests (schema, loader with all four resolution paths, fail-closed on every invalid shape, evaluator covering every default rule, determinism, first-match-wins via reordering, `risk_min` ordinality, `tool:` glob) — total suite 105/105. Manual smoke through built bin: `read_file → stripe_create_charge → read_file` against a counting mock server; the mock reports `total=1` then `total=2` (only the two reads reached it) while the client received `{error: {code: -32010, message: "denied by rule 'never-spend-money' …"}}` for the payment, and the audit log records `seq=1 DENY stripe_create_charge payment/spend risk=critical rule=never-spend-money 0ms`. Invalid policy YAML (`on_unmatched: approve`) makes `wrap` exit 3 with `Expected 'allow' | 'deny' | 'ask' | 'log', received 'approve'`.

## Phase 5 — Approval channels (✅ shipped)

`ask` verdicts now actually pause the call until a human resolves them, *without blocking other traffic* on the relay. The interceptor returns a `pending: Promise<InterceptorResolution>` from a new field on `InterceptorDecision`; the proxy applies the immediate empty decision right away (so the upstream chain advances) and writes the resolved forward/error back to the client only when the promise settles. This is the load-bearing concurrency primitive — multiple asks queue independently and a plain `read_file` interleaved between two pending `shell_exec` asks round-trips immediately. `src/approval/queue.ts` is the `ApprovalQueue` class: keyed by 12-char hex id, exposes `enqueue → {summary, promise}`, `resolve(id, "approve"|"deny", resolvedBy)`, `resolveFromDisk(id)` (out-of-band channel), `list()` for the operator UX, and a default 60s timeout → `deny` with `resolvedBy="timeout"` and `unref`-ed timers so the process can exit naturally. Every enqueue drops a `<id>.json` summary into `~/.airlock/pending/` (mode 0600) so out-of-band tooling can see what's waiting; resolution unlinks both the summary and any `<id>.resolution` file. `src/approval/cli.ts` is the interactive `@inquirer/prompts` confirm channel that runs on stderr (so it doesn't corrupt the stdio JSON-RPC stream on stdout) with a serial prompt chain — disabled with `--no-cli-approval` or when stderr isn't a TTY. `src/approval/notify.ts` is the `node-notifier` desktop ping channel (disabled with `--no-notify` or `AIRLOCK_NO_NOTIFY=1`); both modules use lazy dynamic imports so server-only deployments don't fail on missing native bindings. `src/approval/watcher.ts` runs `fs.watch` + a 1s polling fallback over the pending dir so `<id>.resolution` files written by `airlock approve <id>` / `airlock deny <id>` from another terminal route to the running proxy. The auditing interceptor now records `verdict`, `resolvedBy`, `waitedMs`, and the final `latencyMs` per call; the human view in `airlock log` surfaces `waited=Xms by=cli` so post-hoc audits can see who decided what and how long it took. New CLI commands: `airlock approve <id>`, `airlock deny <id>`, `airlock pending`. Gate ran clean: 14 new tests (10 queue + 4 interceptor ask-path integration including approve→forward, deny→MCP error, timeout→deny, and a concurrency test proving three concurrent asks + an interleaved read all resolve correctly without blocking) — total suite 119/119. Manual smoke through built bin: with a 10s timeout, sending shell_exec/read_file/shell_exec produces the read response immediately while 2 entries appear in `~/.airlock/pending/`; writing `<id>.resolution = approve` to the first and `deny` to the second resolves both, the client sees `id=1 OK`, `id=3 ERROR ...denied by out-of-band`, and `airlock log` records `seq=2 DENY ... waited=449ms by=out-of-band 449ms`, `seq=3 ALLOW ... waited=449ms by=out-of-band 453ms`. With `--approval-timeout-ms 500`, a shell_exec sent and left alone times out and the client receives `airlock: denied by rule 'confirm-shell-exec' (human approval timed out)`.
